# 字节邮箱前缀 + 色卡密码登录（Supabase）设计稿

**项目**：天水围的日与夜（GitHub Pages 前端 + Supabase 云端）

**目标**：
- 让朋友使用：账号隔离（每人只看到自己的 Todo）
- 不依赖短信/邮箱验证码（避免 rate limit 与回跳问题）
- 登录弹窗视觉与站点日/夜风格一致，质感高级
- 忘记密码不做找回，提供“重新注册（同名覆盖）”能力（可控但高风险，必须强提示）

---

## 结论（产品侧）

### 登录方式
- 账号：仅允许字节邮箱前缀，最终规范化为 `${prefix}@bytedance.com`
- 密码：从 9 张色卡中选择 2 张（顺序无关）作为“色卡密码”
- 默认态：登录；提供“没有账号？去注册”入口

### 忘记密码策略
- 不提供找回（无邮件/短信验证通道）
- 提供“同名重新注册（覆盖旧账号）”
  - 覆盖意味着：旧账号会被删除，旧云端 Todo 会被清空
  - 需要二次确认 + 倒计时 + 明确告知风险

---

## 结论（技术侧）

### Auth（Supabase）
- 使用 Supabase 原生 Email + Password
  - 登录：`signInWithPassword({ email, password })`
  - 注册：`signUp({ email, password })`
- 不使用：
  - 手机号 OTP
  - 邮箱 magic link / email OTP（避免限流与跨设备回跳问题）
- 前端继续使用 publishable key + RLS，保证云端数据隔离

### 覆盖注册（同名重置）
Supabase 不允许同一个 email 重复注册。要实现“同名重新注册”，需要 Supabase Edge Function（service role）执行管理动作：
- 输入：`prefix` 与新的“色卡密码”
- 行为：
  1) 查找并删除旧用户（按 email）
  2) 删除该用户的 todos（若 DB 侧已设置 cascade/或显式删除）
  3) 创建新用户（同 email + 新 password）
  4) 返回新用户的 session（或临时密码登录所需信息），让前端直接进入已登录状态

安全提醒：
- 该能力等同“自助清空账号数据”，任何知道别人邮箱前缀的人都能破坏其账号
- 必须在 UI 上做强确认，降低误操作
- 仅建议用于小范围朋友内测；公开场景不建议上线

## 账号规范化规则

### 前缀输入
- 用户只输入前缀（不允许输入 `@`）
- 前缀允许字符：`a-z 0-9 . _ -`
- 长度：2～32
- 最终 email：`${prefix}@bytedance.com`

### 色卡密码（2 色顺序无关）

色卡集合固定为 9 个（全局一致）：
- 以 id 标识：`C1..C9`
- 以颜色值渲染（RGB/hex）

用户选择 2 个色卡 id：`[A, B]`，顺序无关：
- 规范化：对 id 做稳定排序，得到 `[min, max]`
- 生成 Supabase password：`${min}-${max}`
  - 示例：`C2-C8`
  - 长度满足 Supabase 最低密码要求（>= 6）

展示：
- “已选：蓝 + 紫”（不展示顺序）

风险说明（可在产品文案中弱提示）：
- 组合数为 C(9,2)=36，属于弱口令；依赖登录限流与社交圈可控

## UI 设计（弹窗）

### 结构
- 顶部：标题（登录/注册）+ 关闭按钮
- 中部：账号输入（前缀输入框 + 固定后缀 pill）+ 色卡选择器（3x3）
- 底部：主按钮（登录 / 注册并登录）+ 次入口（切换登录/注册）+ 风险提示（仅覆盖注册时）

### 日/夜主题一致性
- 弹窗背景与边框使用 CSS 变量与 `data-theme` 配合：
  - 日：浅色雾面 + 轻阴影
  - 夜：深色玻璃 + 更强景深阴影
- 色卡选中态需要在日/夜都清晰：
  - 选中：描边增强 + 轻微放大 + 内发光
  - 禁用：降低饱和度与透明度

### 行为细节
- 登录/注册中：按钮显示 loading 文案并禁用
- 错误展示：按钮下方以红色文案显示（不依赖 toast 一闪而过）
- 色卡选择规则：
  - 未选满：点击即加入
  - 已选满 2 个：点击第三个会替换最早选择的那个（避免“点了没反应”）

## 功能下线范围

### 下线内容（UI 与入口）
- 登录弹窗中移除：
  - 手机号登录
  - 邮箱验证码/魔法链接登录
- 下线跨设备配对登录入口：
  - 移除 `/#/pair` 路由入口与任何 UI 引导

### 保留内容（可选）
- Supabase 侧 `pair-login` function 与 `pair_logins` 表可以先保留不动（不再被前端调用）

---

## 验收标准

- 未登录可浏览；所有写操作会要求登录
- 登录/注册成功后：
  - Todo 改为云端读写
  - 刷新页面后仍保持登录态与云端 Todo
- 不同账号互相隔离（RLS 生效）
- 弹窗在日/夜主题下视觉一致、无突兀配色
- 覆盖注册触发时：
  - 明确提示“将清空旧云端数据”
  - 必须二次确认与倒计时，误触概率低
